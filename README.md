Project Architecture
The project serves as the underlying unhosted operating system layer designed to execute the Synchrona Graphics Engine directly on bare metal without a standard host OS (such as Windows, Linux, or macOS).

The Kernel: Written in C++ and assembly, handling low-level ARM64 initialization, MMU paging, and direct hardware memory mapping.

The Driver Layer: Fast-path abstractions designed to interface directly with GPU hardware pipelines, bypassing standard operating system scheduling delays.

Core Technical Features
Bare-Metal ARM64 Bootloader: Custom assembly boot sequence initializing CPU cores directly from exception level 3 (EL3/EL2) to EL1.

Unified Memory Paging: Static and dynamic identity mapping configured to utilize unified CPU/GPU memory pools without PCIe bus serialization.

Zero-Overhead Scheduler: A cooperative, single-tasking execution loop designed to give the rendering pipeline 100% of CPU execution time.

Interrupt Optimization: Minimal ISR (Interrupt Service Routine) configuration to ensure zero latency spikes during frame submission.

Direct Hardware Address Spacing: Explicit memory-mapped I/O (MMIO) configurations optimized for high-bandwidth interconnects like NVLink-C2C.

Target Hardware Architecture
The kernel is architected to target high-efficiency, high-bandwidth ARM64 silicon:

Development Platform: Raspberry Pi 5 / QEMU ARM64 Emulation (Current).

The Synchrona Core Kernel is optimized strictly for graphics compute, eliminating the background context-switching common in general-purpose operating systems.

CPU Utilization: By removing background daemons, security layers, and complex user-space subsystems, CPU overhead is held at a absolute minimum (~4%).

Latency Elimination: Standard operating systems introduce micro-stuttering via thread scheduling. This kernel locks execution to the render graph, ensuring flat-line frame times.

# Synchrona Core Kernel
A low-level, bare-metal ARM64 microkernel architected specifically to eliminate operating system overhead and maximize throughput for unified-memory graphics hardware.

---

### Project Architecture
The project serves as the underlying unhosted operating system layer designed to execute the Synchrona Graphics Engine directly on bare metal without a standard host OS (such as Windows, Linux, or macOS).

*   **The Kernel**: Written in C++ and assembly, handling low-level ARM64 initialization, MMU paging, and direct hardware memory mapping.
*   **The Driver Layer**: Fast-path abstractions designed to interface directly with GPU hardware pipelines, bypassing standard operating system scheduling delays.

### Core Technical Features
*   **Bare-Metal ARM64 Bootloader**: Custom assembly boot sequence initializing CPU cores directly from exception level 3 (EL3/EL2) to EL1.
*   **Unified Memory Paging**: Static and dynamic identity mapping configured to utilize unified CPU/GPU memory pools without PCIe bus serialization.
*   **Zero-Overhead Scheduler**: A cooperative, single-tasking execution loop designed to give the rendering pipeline 100% of CPU execution time.
*   **Interrupt Optimization**: Minimal ISR (Interrupt Service Routine) configuration to ensure zero latency spikes during frame submission.
*   **Direct Hardware Address Spacing**: Explicit memory-mapped I/O (MMIO) configurations optimized for high-bandwidth interconnects like NVLink-C2C.

---

### Target Hardware Architecture
The kernel is architected to target high-efficiency, high-bandwidth ARM64 silicon:
*   **Development Platform**: Raspberry Pi 5 / QEMU ARM64 Emulation (Current).
*   **Target Research Hardware**: NVIDIA Grace CPU architecture (RTX Spark / DGX Spark platform).

---

### Comparative Efficiency Analysis
The Synchrona Core Kernel is optimized strictly for graphics compute, eliminating the background context-switching common in general-purpose operating systems.

*   **CPU Utilization**: By removing background daemons, security layers, and complex user-space subsystems, CPU overhead is held at an absolute minimum (~4%).
*   **Latency Elimination**: Standard operating systems introduce micro-stuttering via thread scheduling. This kernel locks execution to the render graph, ensuring flat-line frame times.

---

### Compilation and Deployment
The kernel uses a dedicated cross-compilation toolchain for ARM64 bare-metal targets.

**Prerequisites**
*   `aarch64-none-elf-gcc` / `aarch64-none-elf-g++`
*   CMake build system

**Manual Build from Source**
*   Clone the repository.
*   Initialize the cross-compilation CMake toolchain file targeting `aarch64`.
*   Compile to generate the raw binary image (`kernel8.img`).

**Automated Build Scripts**
*   **Windows**: Execute `build_kernel.bat` to initiate cross-compilation via LLVM/Clang.
*   **Linux/macOS**: Execute `build_kernel.sh` to compile via the GCC ARM64 toolchain.

*   **Unified Memory Paging**: Fine-tuning memory address translation to exploit 128GB unified memory setups.
*   **Ray Query Space**: Allocating dedicated static memory blocks for hardware-accelerated acceleration structure traversal.
